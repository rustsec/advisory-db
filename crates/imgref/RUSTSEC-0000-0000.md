# Out-of-bounds access in iterators with malformed image dimensions

Affected versions of `imgref` did not consistently validate that the image
dimensions and stride described a region fully contained in the backing buffer
before constructing row and pixel iterators.

The crate provides safe constructors such as `Img::new` and `Img::new_stride`
that accept a buffer together with width, height, and stride. If code using
`imgref` created an image from dimensions that could be controlled by untrusted
input, then a too-small backing buffer or overflowing size calculation could
allow safe APIs to reach unsafe iterator code with invalid assumptions.

This could cause out-of-bounds memory access in several ways:

- `RowsIter` used unchecked slice creation for each row.
- `PixelsRefIter` and `PixelsIterMut` performed raw pointer arithmetic while
  walking rows.
- In release builds, unchecked arithmetic in the minimum-buffer-length
  calculation could wrap, causing validation to accept impossible image layouts.

These cases can trigger undefined behavior from safe Rust code. Depending on
how the crate is used, the consequence may include out-of-bounds reads or writes.

Version `1.12.1` added more uniform invariant checks, but the fix was
incomplete because the length calculation could still overflow. Version
`1.12.2` completes the fix by using checked arithmetic when validating image
layout before constructing the affected iterators.

Users should upgrade to `imgref` `1.12.2` or later. If upgrading is not
immediately possible, callers should validate untrusted dimensions before
constructing or iterating over an `Img`: `stride` must be nonzero and at least
`width`, and the backing buffer must contain at least
`stride * (height - 1) + width` elements for non-empty images, with that
calculation performed using checked arithmetic.
